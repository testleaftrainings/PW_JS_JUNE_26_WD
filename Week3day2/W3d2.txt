Assertions:
-----------

Assertions are used to verify that the application is working as expected.
Playwright provides a set of built-in assertions that can be used to check the state of the application.
Assertions can be used to check for the presence of elements, the text content of elements, the value
of input fields,and more.
 
types of Assertions:

1) Generic Assertions : 
----------------------

Generic Assertions are used to validate simple values, variables, calculations, API response fields,
boolean conditions, etc.
These are immediate checks (non-retry) and do NOT depend on UI elements.

Use cases:
---------
Comparing numbers
Comparing strings
Checking boolean results
Validating object values
Checking array lengths
Validating API response data

Common Generic Assertions

1. toBe()

Checks strict equality (number, boolean, etc.)

expect(status).toBe("assigned");
expect(isLoggedIn).toBe(true);

2. toEqual()

Checks value equality for objects or arrays

expect([1, 2]).toEqual([1, 2]);

3. toBeTruthy()

Passes if the value is any “truthy” value.

expect("hello").toBeTruthy();
expect(1).toBeTruthy();

4. toBeFalsy()

Passes if the value is “falsy”.
Falsy values: 0, null, undefined, false, “”

expect("").toBeFalsy();
expect(0).toBeFalsy();

5. toBeGreaterThan()

Checks if a number is greater than another.

expect(10).toBeGreaterThan(5);

6. toBeLessThan()

Checks if a number is less than another.

expect(3).toBeLessThan(10);

7. toBeGreaterThanOrEqual() / toBeLessThanOrEqual()

Useful for ranges.

8. toContain()

Checks if an array or string contains something.

expect([1,2,3]).toContain(2);
expect("playwright").toContain("write");

9. toMatch()

Matches a string using a regex pattern.

expect("Welcome123").toMatch(/Welcome/);

10. toBeNull() / toBeDefined() / toBeUndefined()

State checks.

expect(value).toBeNull();
expect(name).toBeDefined();


2. Locator Assertions:
-----------------------

Locator Assertions are used to validate the state, visibility, text, attributes, and behavior of UI
elements in the webpage.

Locator assertions are AUTO-RETRY, meaning Playwright keeps checking the condition until it becomes 
true or times out.This makes them very reliable for dynamic UI.

Use Cases of Locator Assertions 
-------------------------------

Locator Assertions are used for:

Checking visibility of elements
Verifying buttons are enabled/disabled
Validating text content
Checking input field values
Ensuring elements appear/disappear
Counting number of elements

Common Locator Assertions
--------------------------

1. toBeVisible()

Checks that the element is visible on the page.
await expect(locator).toBeVisible();

2. toBeHidden()

Checks that the element is present but not visible.
await expect(locator).toBeHidden();

3. toBeEnabled()

Checks if the element is enabled and interactive.
await expect(button).toBeEnabled();

4. toBeDisabled()

Checks if the element is disabled (cannot be clicked or typed into).
await expect(button).toBeDisabled();

5. toBeEditable()

Checks if an input field is editable by the user.
await expect(input).toBeEditable();

6. toHaveText()

Checks for exact match of text inside an element.
await expect(locator).toHaveText("Login Successful");

7. toContainText()

Checks if the element contains partial text (substring).
await expect(locator).toContainText("Success");

8. toHaveValue()

Checks the value of input elements (textbox, textarea).
await expect(input).toHaveValue("Testleaf");

9. toHaveAttribute()

Checks if an element has a specific attribute.
await expect(link).toHaveAttribute("href", "/dashboard");

10. toHaveCount()

Checks how many elements match the locator.
await expect(locator).toHaveCount(5);

11. toBeChecked() / toBeUnchecked()

Used for checkboxes, radio buttons, switches.
await expect(checkbox).toBeChecked();

Examples:

Check if the login button is visible
Verify the error message contains correct text
Validate username field is editable
Ensure product list displays 10 items
Check the checkbox is selected after clicking it

Locator assertions auto-retry because UI loads dynamically.
This is why they are more stable in automation.

 Generic Assertions = value checks
 Locator Assertions = UI element check


3. Page Assertion.

Assertions made on the page object , such as URL and title.

toHaveURL(url) - checks if the current page URL matches the expected URL.
toHaveTitle() - checks if the page title matches the expected title.

4. Snapshot Assertions. --> Visual Regression testing
Snapshot assertions are used for visual regression testing by comparing the current UI screenshot 
with a previously saved snapshot to detect visual changes from the UI perspective.

5. API refernce Assertions. --> REST API 
Assertions used check your API reponses 
expect(response).toHaveStatus(200) // Checks if the response status code is 200

Based on Functionality 
--------------------

1. Auto retry assertion.
2. Non-retry assetion.


Hard Assertion.--> It will abruptly stop your execution.
Soft Assertion. --> Informing that the assertion has failed and it will not stop execution it will
move to next part of execution.

default assertion is hard assertion in playwright.
timeout for hard assertion is 5000ms and for soft assertion is 0ms.
we can change the default timeout for hard assertion by using below code in config file.
expect:{    
timeout:10000
}

timeouts in playwright :
----------------------

1. test timeout:
-----------------
Timeout for a specific test execution.

test.setTimeout(20000)

2. assertion timeout:
----------------------
Timeout for assertion of a weblelement.

In config file below timeout :
expect:{
timeout:15000
}

In script : 
await expect(locator).toHaveText('Submit',{timeout:8000})

3. Action timeout :
--------------------
Timeout for each and every action performed like click(), fill()

In config mention under "use" :
use:{
actionTimeout:15000
}

In script : 
await page.locator("#password").fill("crmsfa",{timeout:2000})

4. GLobal timeout :
-----------------------
Implemented when we execute in pipleines.

In config file under "timeout" :

globalTimout : 60*60*1000

| ---------------------- | ----------------------------------- |
| Checks structure       | Checks appearance                   |

storageState:
-------------

Storage State is a Playwright feature used to save and restore a browser context's authentication 
data, such as cookies and local storage, into a JSON file. This allows tests to reuse an existing
user session, bypass the login process, and start with an authenticated session without logging in 
again.

steps:
------
Stores cookies and local storage data.
Saved as a JSON file (e.g., auth.json).
Helps reuse authenticated sessions across tests.
Eliminates the need to perform login repeatedly.
Improves test execution speed and stability.

Browser stores.
 ├─ Cookies -> Used to store authentication/session information shared with the server.
 ├─ Local Storage -> Used to store persistent client-side data across browser sessions.
 ├─ Session Storage-> Used to store temporary tab-specific data until the tab is closed.


// Save authenticated session
await page.context().storageState({
  path: 'auth.json'
});

// Reuse authenticated session

(with Browser instance)
const browser = await chromium.launch(); 
const context = await browser.newContext({ storageState: 'user.json' });
const page = await context.newPage();

or 
(with page fixture)
test.use({
  storageState: 'Data/user.json'
});

Note:Storage State is valid until the stored session cookies or auth tokens expire or become invalid.
